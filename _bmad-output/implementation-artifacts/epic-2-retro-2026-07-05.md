## 2. What Went Well

- **The cross-story intelligence loop kept paying off** (Epic 1's standout asset, sustained). Every story opened with a *Previous-story intelligence* section + an explicit *scope boundary*. 2.2 knew exactly which text tokens the 2.1 review renamed (`--tx*` → `--text-*`); 2.4 knew how 2.2 had deliberately shaped `App.tsx` for the real `fetch`. Zero re-work from inter-story misalignment.
- **The integration-test layer (Epic 1 Action Item #1) rendered real security value.** Built before 2.3, it caught genuine behaviour: session key present/absent in Redis, RBAC array-overlap returning exactly the intersecting channels and not others, per-request recompute (policy change between two calls on the same session changes the result — proving it is not cached).
- **The code review caught concentrated security issues in 2.3 and they were fixed in-story:** session fixation (session id not regenerated after auth), no fetch timeout on Discord calls (callback could hang), unvalidated `access_token` shape, plus **CSRF `state` implemented proactively** beyond the ACs.
- **DDD-by-layer landed clean and extensible.** `authService`/`rbacService` depend only on domain interfaces, unit-testable with plain fakes; `app.ts` is the composition root. Future backend epics (search, documents, chat) extend the same layers.
- **"Done" meant verified against reality.** Manual boot smoke tests exercised the real generic gate (401 on `/api/anything`), startup `channel_permissions` materialization, and exempt `/health` — not just a green build.

## 3. Challenges

- **External-integration assumptions were validated only at implementation time — again.** The whole stack (docs, ARCHITECTURE-SPINE AD-10/AD-13) assumed `connect-redis@9` bound to ioredis. It does not — `connect-redis@9` is node-redis-only. Discovered mid-2.3. The response was correct: because the ioredis footprint was tiny and the bot/workers Streams consumers do not exist yet, the project standardized on **node-redis (`redis@6`)** and corrected AD-10/AD-13 + stack tables across four authoritative docs. This is the **same pattern as Epic 1** (7-vs-8 tables): integration reality surfaces late. **Flagged by Borja as a top concern for Epic 3, which introduces two new external integrations.**
- **Visual/pixel ACs are not automatically verifiable under jsdom.** In 2.1 and 2.2, colour/token/font ACs cannot be asserted (jsdom applies no external CSS and does not resolve custom properties via `getComputedStyle`). Handled honestly ("manual browser check for the reviewer"), but a design-heavy epic ships a band of ACs whose only verification is a human eyeballing pixels.
- **Tooling/config gotchas recurred.** 2.1: `tsc` NodeNext vs Vite Bundler module resolution for relative + CSS imports. 2.2: jsdom test-file discovery (`.ts` node vs `.tsx` web) and no Web Storage under Node 24 + jsdom (stub `localStorage` in the test). Each cost real debug time and is now documented.

## 7. Epic Update Decision

**No re-plan required for Epic 3.** No Epic 2 assumption invalidates the Epic 3 plan — the node-redis pivot actually *de-risks* the Streams consumers (born with the correct client). What remains is a strong **preparation warning** (Action Item #1): Epic 3's two new external integrations (`discord.js` Gateway, embeddings API) are the real risk, and Epic 2 (ioredis) showed exactly how that risk bites when validated late.

## 8. Key Takeaways

1. The cross-story intelligence loop + integration-test-first discipline are the two process assets that produced trustworthy "done" in Epic 2 — keep feeding both, and extend the integration layer to the new Streams/pgvector boundaries in Epic 3.
2. External-integration assumptions surface late (ioredis, 7-vs-8 tables). Epic 3 has two new ones — validate them with a real smoke spike *before* building on top.
3. The code review is a genuine security net (2.3): session fixation, CSRF, timeouts, response-shape validation. Preserve the adversarial review bar for the pipeline's new attack surface.
4. jsdom cannot verify visual ACs — a known, accepted gap for design-heavy work; a lightweight visual-verification approach is worth considering if future UI epics carry more pixel-level ACs.
